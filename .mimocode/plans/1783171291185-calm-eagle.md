# План исправлений — Платёжный календарь

## Итоги аудита

Проект функционален, но содержит критические баги, проблемы безопасности и неполные реализации. Ниже — поэтапный план исправлений, сгруппированных по приоритету.

---

## Этап 1: Критические баги (сломанная логика)

### 1.1 Двойной хеширование паролей — UserController
**Файл:** `backend/app/Http/UserController.php:35,69`
**Проблема:** `Hash::make()` вызывается вручную, но модель User имеет каст `'password' => 'hashed'`, который автоматически хеширует при `create()`/`update()`. Результат — двойной bcrypt, пароль невозможно верифицировать при логине.
**Исправление:** Убрать ручные `Hash::make()` из `store()` (строка 35) и `update()` (строка 69). Модель сама хеширует.

### 1.2 Store позволяет задать статус платежа напрямую — PaymentController
**Файл:** `backend/app/Http/PaymentController.php:51`
**Проблема:** Валидация `'status' => 'sometimes|in:draft,pending,approved,...'` позволяет создать платёж сразу со статусом `approved` или `paid`, обходя весь workflow согласования.
**Исправление:** Убрать `status` из валидации `store()`. Всегда ставить `'draft'`.

### 1.3 Update поступления позволяет менять статус без ограничений — IncomeController
**Файл:** `backend/app/Http/IncomeController.php:94`
**Проблема:** `update()` принимает `status` и не проверяет переходы. Можно перевести `canceled` → `received` в обход `markConfirmed`/`markReceived`/`cancel`.
**Исправление:** Убрать `status` из валидации `update()`. Статус меняется только через dedicated endpoints.

### 1.4 Отклонённый платёж навсегда застревает — PaymentController
**Файл:** `backend/app/Http/PaymentController.php:132-136`
**Проблема:** `submit()` разрешает только `draft` → `pending`. Отклонённый платёж (`rejected`) нельзя ни отредактировать (update запрещает), ни отправить повторно.
**Исправление:** Разрешить `submit()` для статусов `draft` и `rejected`. Разрешить `update()` для статусов `draft` и `rejected`.

## Этап 2: Безопасность и целостность данных

### 2.1 Убрать `$guarded = []` из моделей
**Файлы:** `backend/app/Models/Account.php:12`, `backend/app/Models/Counterparty.php:12`, `backend/app/Models/Payment.php:12`
**Проблема:** `$guarded = []` делает `$fillable` бесполезным и открывает mass-assignment для любых полей.
**Исправление:** Удалить строки `$guarded = []` из всех трёх моделей.

### 2.2 Обработка FK-constraint при удалении
**Файлы:** `AccountController.php:83-91`, `CounterpartyController.php`, `ItemController.php`, `UserController.php:83-91`
**Проблема:** Удаление счёта/контрагента/статьи/пользователя, на который ссылаются платежи, выбрасывает необработанную 500-ку.
**Исправление:** Обернуть `delete()` в try-catch, возвращать 422 с понятным сообщением.

### 2.3 RegistryController::pay() — нет проверки на `canceled`
**Файл:** `backend/app/Http/RegistryController.php:93-97`
**Проблема:** Проверяется только `paid`, но `canceled` реестр тоже можно оплатить.
**Исправление:** Добавить проверку: если `status !== 'created'` → 422.

### 2.4 Транзакции в multi-record операциях
**Файлы:** `RegistryController.php:58-68` (store), `RegistryController.php:99-106` (pay), `PaymentController.php:62-84` (recurring)
**Проблема:** Нет `DB::transaction()`. При ошибке на середине — частичное состояние.
**Исправление:** Обернуть в `DB::transaction()`.

## Этап 5: Импорт — исправления парсинга

### 5.1 Парсинг даты без zero-padding
**Файл:** `backend/app/Http/ImportController.php:50`
**Проблема:** `5.6.2026` → `2026-6-5` без ведущих нулей.
**Исправление:** Использовать `sprintf("%04d-%02d-%02d", ...)`.

### 5.2 Парсинг суммы — нет обработки десятичных
**Файл:** `backend/app/Http/ImportController.php:59`
**Проблема:** `1500.50` → `150050` (×100). `1 500,00` → `150000` (корректно только если нет копеек).
**Исправление:** Разделить на рубли и копейки, корректно конвертировать в копейки.

## Верификация

1. `php artisan migrate:fresh --seed` — БД пересоздаётся без ошибок
2. Логин через API (`POST /api/login`) — токен получается (проверяет двойной хеш)
3. Создание платежа через API — статус всегда `draft`
4. Workflow: draft → pending → approved → in_registry → paid
5. Rejected → повторный submit → pending → approved
6. `GET /api/calendar?start_date=...&end_date=...` — остатки по дням, кассовые разрывы
7. `POST /api/registries` + `POST /api/registries/{id}/pay` — реестр создаётся и оплачивается
8. `GET /api/audit` — записи логов появляются после операций
9. Фронтенд: фильтры счетов загружаются из API, мёртвый код удалён
