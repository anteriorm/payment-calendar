Полная функциональная карта фронтенда
1. Архитектура приложения
Стек: React (функциональные компоненты + хуки), React Router v6, Axios, управление состоянием (React Context + useReducer или Redux Toolkit — на ваше усмотрение), UI-библиотека (Ant Design / Material-UI / Mantine).

Структура папок:

├── contexts/       # Контексты для авторизации, уведомлений, темы

Управление состоянием:

Глобальное: текущий пользователь, роли, уведомления, список справочников (кеш).

Локальное: состояния страниц (фильтры, пагинация, загрузка, ошибки, модальные окна).

2. Авторизация и управление сессией
Страница /login

Форма: email, password.

Отправка POST /api/login.

При успехе сохраняем токен в localStorage (или httpOnly cookie – но для простоты localStorage).

Сохраняем данные пользователя (id, name, email, role) в контекст.

Редирект на /dashboard.

Страница /logout (кнопка в шапке)

Вызов POST /api/logout (опционально, но лучше делать).

Удаление токена и данных пользователя.

Редирект на /login.

Защита маршрутов

Создать PrivateRoute – проверяет наличие токена, иначе редирект на логин.

Создать RoleRoute – проверяет роль пользователя, иначе показывает 403 или редирект.

Обновление токена (если реализовано на бекенде) – можно добавить перехватчик 401 с автоматическим обновлением, но для MVP достаточно просто редиректа на логин.

3. Главная страница / Дашборд (/dashboard)
Цель – дать пользователю быстрый обзор текущей ситуации.

Карточки-виджеты:

Общий остаток по всем счетам (сумма closing_balance на сегодня или на конец периода).

Ближайший кассовый разрыв (дата и сумма отрицательного остатка).

Количество заявок со статусом pending (ожидают согласования).

Сумма платежей на сегодня.

Сумма поступлений на сегодня.

График (опционально): простая столбчатая диаграмма остатков по дням за неделю.

Источник данных: отдельный эндпоинт /api/dashboard или агрегация из календаря и заявок.

4. Модуль «Справочники»
Все справочники имеют одинаковую структуру: таблица + кнопка «Создать» + модальное окно (или отдельная страница) для создания/редактирования.

4.1. Счета (/accounts)
Поля: Название, Тип (банк/касса), Валюта (RUB), Начальный остаток (в рублях, но в API передаём в копейках).

Функции: CRUD.

Валидация: название обязательно, остаток – число >= 0.

Использование: счета выбираются в формах платежей и поступлений.

4.2. Контрагенты (/counterparties)
Поля: Название, ИНН, КПП (опционально), Расчётный счёт, Банк, БИК.

Функции: CRUD.

Валидация: название обязательно, ИНН – число (можно проверить длину).

Использование: выбор контрагента в платежах и поступлениях.

4.3. Статьи движения денег (/items)
Поля: Название, Тип (income или payment).

Функции: CRUD.

Фильтрация: при создании платежа показываем только payment, при поступлении – income.

Общие требования для всех справочников:

Пагинация (или бесконечный скролл, если данных много).

Поиск/фильтр по названию.

Подтверждение удаления (модалка «Вы уверены?»).

Отображение ошибок валидации (422) рядом с полями.

5. Модуль «Заявки на платёж» (/payments)
Это ключевая сущность для списаний.

Страница списка:

Таблица с колонками: ID, Дата, Счёт, Контрагент, Статья, Сумма, Приоритет, Статус, Назначение.

Фильтры:

По статусу (мультивыбор).

По счёту (выпадающий список).

По контрагенту (поиск).

По периоду (дата с – по).

По приоритету (высокий/средний/низкий).

Сортировка по дате, сумме.

Кнопка «Создать заявку» → открывается форма создания.

Форма создания/редактирования:

Поля: Счёт (select), Контрагент (select), Статья (select с фильтром payment), Сумма (input), Дата (DatePicker), Приоритет (radio/select), Назначение (textarea).

Для редактирования форма предзаполнена.

Валидация: все поля обязательны, сумма > 0, дата не в прошлом (опционально).

При сохранении: POST /api/payments (создание) или PUT /api/payments/{id} (редактирование).

После сохранения – обновление списка.

Действия над заявкой (зависят от статуса и роли):

Инициатор:

Для draft: редактировать, удалить, отправить на согласование (POST /api/payments/{id}/submit).

Для pending: нельзя редактировать (только смотреть).

Для rejected: можно отредактировать и отправить заново (если разрешено).

Руководитель:

Для pending: кнопки «Согласовать» (POST /api/payments/{id}/approve) и «Отклонить» (POST /api/payments/{id}/reject). При отклонении – поле для комментария.

Казначей:

Для approved: кнопка «Включить в реестр» (но это делается на странице реестра).

Для in_registry и paid – только просмотр.

Администратор: может менять статусы принудительно (опционально).

История согласований: можно добавить вкладку или модалку с логом (запрос GET /api/payments/{id}/approvals).

6. Модуль «Поступления» (/incomes)
Аналогичен платежам, но без сложного workflow.

Страница списка:

Таблица: Дата, Счёт, Контрагент, Статья, Сумма, Назначение, Статус (planned, received, canceled).

Фильтры: по статусу, счёту, контрагенту, периоду.

Действия:

Создать поступление.

Редактировать (если статус planned).

Удалить (если planned).

Отметить как «Получено» (POST /api/incomes/{id}/received) – статус меняется на received.

Отменить (POST /api/incomes/{id}/cancel) – статус canceled.

Форма создания/редактирования аналогична платежам, но тип статьи – income.

Интеграция с календарём: поступления учитываются при расчёте остатков (кроме canceled).

7. Модуль «Платёжный календарь» (/calendar)
Это главный экран, ради которого всё затевалось.

Верхняя панель фильтров:

Период: два DatePicker – «Дата начала» и «Дата конца» (по умолчанию текущий месяц).

Счёт: выпадающий список (можно выбрать «Все счета»).

Статус платежа (для отображения): возможно, фильтр по статусам, но по умолчанию все кроме draft и rejected.

Таблица календаря:

Колонки:

Дата (день месяца, например, «01.07.2026»).
Счёт (если выбран «Все», то для каждого счёта своя строка, либо сводная строка по всем счетам).
Остаток на начало дня.
Поступления за день (сумма).
Платежи за день (сумма).
Остаток на конец дня.
Индикатор кассового разрыва (красная иконка или красный фон строки).
Если closing_balance < 0, строка подсвечивается красным.

Возможность «развернуть» день – показать список платежей и поступлений, которые попали в этот день (с ссылкой на карточку платежа).

Действия в календаре:

Перенос платежа: возле каждого платежа в развёрнутом списке – кнопка «Перенести» (или иконка календаря). Открывается модалка с выбором новой даты. После подтверждения отправляется POST /api/payments/{id}/move с новой датой, календарь перезагружается.

При клике на сумму (или отдельную ячейку) можно открыть детализацию.

Сводка по всем счетам: можно добавить отдельную строку «Итого по всем счетам» в конце таблицы.

Обновление данных: при изменении фильтров, после переноса, после создания/редактирования платежа – автоматически перезапрашивать календарь.

Источник данных: GET /api/calendar с параметрами start_date, end_date, account_id (опционально).

8. Модуль «Кассовые разрывы» (специальный виджет/отчёт)
Хотя кассовые разрывы подсвечиваются в календаре, можно сделать отдельную страницу или виджет, где собраны все дни с отрицательным остатком за выбранный период.

Страница /cash-gaps (или вкладка в отчётах):

Фильтры: период, счёт.

Таблица: Дата, Счёт, Отрицательный остаток.

Действие: кнопка «Показать в календаре» – переход на календарь с этим днём.

9. Модуль «Реестры платежей» (/registries)
Страница списка реестров:

Таблица: ID, Дата реестра, Статус (draft/approved/paid/canceled), Кто создал, Кто утвердил, количество платежей, общая сумма.

Кнопка «Создать реестр».

Создание реестра (пошагово или модалка):

Выбор даты реестра (registry_date).

Загрузка всех платежей со статусом approved на эту дату (или до этой даты) – можно сгруппировать по счёту.

Отображение списка с чекбоксами для выбора.

Сортировка по приоритету (высокий – сверху).

Показ доступного остатка на счёте (предупреждение, если сумма выбранных платежей превышает остаток).

Кнопка «Сформировать реестр» – отправка POST /api/registries с выбранными payment_ids и registry_date. После создания статус платежей становится in_registry.

Просмотр реестра:

Отдельная страница /registries/{id} или модалка.

Список платежей в реестре.

Кнопка «Утвердить реестр» (для руководителя) – меняет статус реестра на approved (если требуется).

Кнопка «Оплатить» (для казначея) – POST /api/registries/{id}/pay – статус реестра становится paid, все платежи внутри – paid.

Кнопка «Экспорт CSV» – скачивание файла (ссылка на GET /api/registries/{id}/export).

10. Модуль «Отчёты» (/reports)
Страница с вкладками или отдельными роутами для каждого отчёта.

Отчёт 1: Остатки по счетам

Фильтры: период, счёт.

Таблица: Дата, Счёт, Остаток на конец дня.

Кнопка «Экспорт CSV» – выгрузка.

Отчёт 2: Кассовые разрывы

Фильтры: период, счёт.

Таблица: Дата, Счёт, Отрицательный остаток.

Экспорт CSV.

Отчёт 3: План-факт по поступлениям и платежам

Группировка по статьям или контрагентам.

Сравнение плановых сумм (все income/payment с учётом статусов) и фактических (со статусом received для поступлений и paid для платежей).

График (столбчатая диаграмма) – опционально.

Техническая реализация: все отчёты – это GET-запросы к соответствующим эндпоинтам с параметрами. Ответ в JSON, фронт рендерит таблицу. Экспорт – отдельный GET, который возвращает CSV (можно открыть в новой вкладке или использовать Blob).

11. Модуль «Импорт/Экспорт» (дополнительно)
Импорт из файла (CSV/Excel)

Страница /import или модалка.

Загрузка файла с колонками: type, date, amount, account_name, counterparty_name, item_name, purpose и т.д.

Предпросмотр данных (первые 5 строк).

Валидация перед отправкой.

Отправка на бекенд (возможно, через POST /api/import).

Отображение результата (сколько создано, какие ошибки).

Экспорт календаря или платежей – можно добавить кнопку «Экспорт» на странице календаря, которая скачивает CSV с данными за выбранный период.

12. Модуль «Повторяющиеся платежи» (дополнительно)
При создании платежа добавить чекбокс «Повторяющийся».

Если включён, появляются поля:

Периодичность (daily, weekly, monthly, yearly).

Дата окончания (recurrence_until).

При отправке на бекенд создаётся не один платёж, а серия (бекенд генерирует их автоматически). Фронт просто передаёт флаг и параметры.

В списке платежей можно отображать иконку «повторяющийся» и возможность просмотреть все созданные экземпляры.

13. Модуль «Аудит действий» (для администратора)
Страница /audit (доступна только admin).

Таблица с логом: Дата и время, Пользователь, Действие (payment_created, payment_status_changed, registry_created и т.д.), Объект (тип и ID), Старые и новые значения (в удобном формате).

Фильтры: по пользователю, по типу объекта, по периоду.

Ссылка на просмотр изменённого объекта (переход к карточке платежа и т.п.).

Интеграция: бекенд должен предоставлять GET /api/audit с пагинацией и фильтрами.

14. Управление пользователями (администратор)
Страница /users (только admin).

Список пользователей: ID, Имя, Email, Роль.

Создание нового пользователя (форма с полями name, email, password, role).

Редактирование (смена роли, сброс пароля).

Удаление.

Эндпоинты: GET /api/users, POST /api/users, PUT /api/users/{id}, DELETE /api/users/{id}.

15. Общие компоненты и утилиты
Таблицы: переиспользуемый компонент с пагинацией, сортировкой, фильтрами.

Модальные окна: для создания/редактирования, подтверждения действий.

Уведомления: глобальный тост-контейнер (например, react-hot-toast или Ant Design message).

Загрузка: спиннеры, скелетоны.

Обработка ошибок: перехватчик в axios, который показывает сообщение об ошибке из ответа бекенда.

Форматирование:

formatCurrency(amountInKopecks) – возвращает строку вида «1 500,50 ₽».

formatDate(dateString) – «25.07.2026».

getStatusLabel(status) – переводит английский статус в русский.

Права доступа: хук usePermissions() возвращает объект с функциями canCreatePayment(), canApprove(), canManageRegistries() и т.д., основанными на роли.

17. Технические детали реализации
Маршрутизация: использовать createBrowserRouter с вложенными маршрутами.

Состояние фильтров: сохранять в URL (query-параметры), чтобы можно было делиться ссылками.

Пагинация: реализовать через query-параметры page, per_page.

Кеширование: для справочников можно закешировать ответы на уровне приложения (например, через React Context), чтобы не дёргать API при каждом открытии формы.

Валидация форм: использовать библиотеку react-hook-form с yup или zod для схем.

DatePicker: использовать компонент из UI-библиотеки, обязательно с локализацией на русский.

CSV-экспорт: либо прямой GET, который возвращает файл (использовать window.open или <a download>), либо генерировать CSV на фронте из данных (но проще доверить бекенду).

Drag-and-drop (опционально): можно добавить перетаскивание платежей в календаре для переноса даты (использовать react-dnd или @dnd-kit).

18. Обработка ошибок и состояний
Loading: показывать спиннер на кнопке или скелетон для таблицы.

Empty state: если нет данных – показывать сообщение «Нет записей» с предложением создать.

Error state: если запрос упал – показывать ошибку с кнопкой «Повторить».

Сетевая ошибка: перехватывать и показывать «Нет связи с сервером».

19. Адаптивность
Хотя проект скорее десктопный, стоит убедиться, что таблицы не ломаются на экранах 1366x768. Для мобильных устройств можно ограничиться базовой адаптацией (скрытие колонок, горизонтальный скролл).

20. Подготовка к демонстрации
Залить тестовые данные через сиды (пользователи, счета, контрагенты, статьи, платежи, поступления).

Подготовить демо-сценарий (можно в README или отдельном документе).

Проверить, что все роли работают, переключение между пользователями без перезагрузки страницы.

Заключение
Это полная картина фронтенда. Вы можете реализовывать модули поэтапно, начиная с MVP, но теперь у вас есть дорожная карта для всего проекта. Помните о главном: календарь – ядро, всё остальное – инструменты для его наполнения и управления.
