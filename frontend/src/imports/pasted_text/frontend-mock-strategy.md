Конечно! Вот чёткое разделение фронтенд-части на обязательный к реализации функционал и то, что можно сделать с заглушками (моками), чтобы бэкендеры могли спокойно разрабатывать свои API, а фронтенд уже был готов к интеграции.

1. Почему заглушки — это правильно
Фронтенд не ждёт бэкенд – вы сразу делаете интерфейс, формы, валидацию, таблицы, фильтры.

Бэкендеры видят готовый интерфейс и знают, какие данные и в каком формате нужно отдавать.

Интеграция происходит плавно – достаточно заменить вызов мока на реальный запрос, и всё работает.

2. Общая стратегия: API-слой с переключением
Вся работа с данными идёт через единый сервис, который возвращает промисы. Внутри него мы используем флаг USE_MOCK (из переменных окружения) для выбора: отдать заглушку или сделать реальный запрос.

Структура:

text
src/
  api/
    client.js          # axios-инстанс
    endpoints/         # папка с модулями-сервисами
      auth.js
      accounts.js
      counterparties.js
      items.js
      payments.js
      incomes.js
      calendar.js
      registries.js
      reports.js
      users.js
      audit.js
    mocks/             # заглушки
      data/            # статические JSON-массивы
        accounts.js
        counterparties.js
        ...
      handlers.js      # функции-заглушки (возвращают промис с данными)
    index.js           # экспорт всех сервисов
  utils/
    config.js          # чтение переменной USE_MOCK
Пример реализации config.js:

javascript
export const USE_MOCK = process.env.REACT_APP_USE_MOCK === 'true';
Пример сервиса accounts.js:

javascript
import api from '../client';
import { USE_MOCK } from '../../utils/config';
import { mockAccounts, mockAccount } from '../mocks/data/accounts';
import { delay, randomId } from '../mocks/handlers';

// Реальные вызовы
const real = {
  getAll: () => api.get('/accounts'),
  getOne: (id) => api.get(`/accounts/${id}`),
  create: (data) => api.post('/accounts', data),
  update: (id, data) => api.put(`/accounts/${id}`, data),
  delete: (id) => api.delete(`/accounts/${id}`),
};

// Заглушки
const mock = {
  getAll: () => delay(mockAccounts),
  getOne: (id) => delay(mockAccounts.find(a => a.id === id) || null),
  create: (data) => delay({ ...data, id: randomId() }),
  update: (id, data) => delay({ ...data, id }),
  delete: () => delay({ success: true }),
};

export const accountsService = USE_MOCK ? mock : real;
Вспомогательные функции handlers.js:

javascript
export const delay = (data, ms = 300) =>
  new Promise(resolve => setTimeout(() => resolve(data), ms));

export const randomId = () => Math.floor(Math.random() * 10000);
3. Что нужно реализовать (без заглушек, чисто фронтенд)
Этот функционал не зависит от данных – он делает интерфейс живым и готовым к работе с любыми данными.

3.1. Авторизация и навигация
Страница входа – форма, валидация (email, пароль), обработка отправки.

Сохранение токена – в localStorage или контекст.

Защита маршрутов – PrivateRoute, RoleRoute.

Layout – шапка с именем пользователя, аватаркой, меню (динамическое, в зависимости от роли).

Выход – очистка токена и редирект.

3.2. Общие UI-компоненты
Таблицы с пагинацией, сортировкой, поиском/фильтрацией.

Модальные окна для создания/редактирования.

Формы с валидацией (использовать react-hook-form + схему).

Компоненты для выбора даты (DatePicker), выпадающие списки (Select), поля ввода суммы с маской.

Уведомления (тосты) об успехе/ошибке.

Индикаторы загрузки (спиннеры, скелетоны).

Подтверждение удаления (диалог).

3.3. Маршрутизация и структура страниц
Все роуты должны быть определены, даже если страница временно пустая.

Боковое меню должно вести на эти роуты.

3.4. Бизнес-логика на фронте (не связанная с данными)
Отображение статусов в понятном виде (русские названия, цветовые индикаторы).

Условный рендеринг кнопок в зависимости от статуса и роли (например, для draft показываем «Отправить», для pending – «Согласовать/Отклонить»).

Пересчёт итоговых сумм в таблицах (например, общая сумма реестра).

Валидация форм: сумма > 0, дата не пустая, обязательные поля.

Форматирование валюты и дат (утилиты).

4. Что нужно реализовать с заглушками (моками)
Здесь мы подготавливаем страницы, полностью рабочие визуально и по логике, но данные берутся из локальных JSON-объектов вместо реального API. Бэкендеры увидят готовый интерфейс и смогут заменить моки на реальные запросы.

Для каждого модуля ниже описано:

Какие страницы/компоненты сделать.

Какие заглушки подготовить (примеры данных).

Какие эндпоинты бэкендер должен будет реализовать, чтобы фронт перестал использовать моки.

4.1. Справочники: Счета, Контрагенты, Статьи
Страницы: /accounts, /counterparties, /items.

Что реализовать полностью:

Таблица с колонками.

Кнопка «Создать», открывающая модалку/форму.

Форма с полями (для счетов: название, тип, валюта, начальный остаток; для контрагентов: название, ИНН, реквизиты; для статей: название, тип).

Валидация: обязательные поля, проверка чисел.

Редактирование (по клику на строку или кнопку).

Удаление с подтверждением.

Фильтр по названию.

Пагинация (можно простую, на стороне фронта, если моки небольшие).

Заглушки:

Массивы объектов с типовыми данными (например, 5 счетов, 10 контрагентов, 8 статей).

Функции-заглушки: getAll, getOne, create, update, delete – возвращают промисы с данными (с задержкой 300 мс).

При создании/редактировании мок должен эмулировать присвоение ID и изменение массива (можно хранить данные в переменной, но после перезагрузки они сбросятся – для прототипа это нормально).

Эндпоинты для замены:

GET /api/accounts

POST /api/accounts

PUT /api/accounts/{id}

DELETE /api/accounts/{id}

Аналогично для /counterparties и /items.

4.2. Заявки на платёж (/payments)
Что реализовать полностью:

Таблица со всеми колонками (ID, дата, счёт, контрагент, статья, сумма, приоритет, статус, назначение).

Фильтры: по статусу (мультиселект), по счёту (выпадающий список), по контрагенту (поиск), по периоду (даты).

Сортировка по дате и сумме.

Кнопка «Создать заявку» → форма с выбором счёта, контрагента, статьи (только типа payment), суммы, даты, приоритета, назначения.

Валидация: все поля обязательны, сумма > 0.

После сохранения – обновление списка.

Для каждой заявки – кнопки действий в зависимости от статуса и роли:

draft: «Редактировать», «Удалить», «Отправить на согласование».

pending: «Согласовать», «Отклонить» (с комментарием).

approved: (ничего, кроме просмотра).

rejected: (можно сделать «Пересоздать» или редактировать).

in_registry, paid: только просмотр.

При отправке, согласовании, отклонении – отправлять соответствующие запросы (заглушки будут эмулировать изменение статуса).

Заглушки:

Массив из 15–20 платежей с разными статусами, датами, счетами.

Функции: getAll (с поддержкой фильтров – можно реализовать простую фильтрацию в памяти), getOne, create, update, delete, submit, approve, reject, move (перенос даты).

При изменении статуса – мок меняет поле status в локальном массиве.

Эндпоинты для замены:

GET /api/payments (с параметрами фильтров)

POST /api/payments

PUT /api/payments/{id}

DELETE /api/payments/{id}

POST /api/payments/{id}/submit

POST /api/payments/{id}/approve

POST /api/payments/{id}/reject

POST /api/payments/{id}/move (body: { planned_date: 'YYYY-MM-DD' })

4.3. Поступления (/incomes)
Что реализовать полностью:

Таблица поступлений с колонками: дата, счёт, контрагент, статья, сумма, назначение, статус.

Фильтры: статус, счёт, период.

Кнопка «Создать» → форма (аналогична платежам, но тип статьи – income).

Редактирование, удаление (только для planned).

Кнопки «Отметить полученным» (статус → received) и «Отменить» (canceled) – с подтверждением.

Заглушки:

Массив поступлений с разными статусами.

Функции getAll, create, update, delete, markReceived, cancel.

Эндпоинты для замены:

GET /api/incomes

POST /api/incomes

PUT /api/incomes/{id}

DELETE /api/incomes/{id}

POST /api/incomes/{id}/received

POST /api/incomes/{id}/cancel

4.4. Платёжный календарь (/calendar)
Что реализовать полностью:

Фильтры: период (два DatePicker), счёт (выпадающий список со значением «все»).

Таблица: строки по дням (каждая строка – день + счёт, если выбран конкретный счёт, или сводная по всем счетам).

Колонки: дата, остаток на начало, поступления, платежи, остаток на конец, индикатор кассового разрыва (красный фон).

Возможность развернуть день (клик на строку) и увидеть список платежей и поступлений за этот день (с суммами и назначениями).

Кнопка «Перенести» рядом с каждым платежом в развёрнутом списке – открывает модалку с выбором новой даты.

После переноса – перезапрос календаря.

Заглушки:

Функция getCalendar(startDate, endDate, accountId) возвращает массив дней с рассчитанными остатками. Чтобы эмулировать кассовые разрывы, в некоторых днях closing_balance должно быть отрицательным.

Данные можно генерировать на основе списка платежей и поступлений (в моке можно реализовать простой расчёт).

Эндпоинт для замены:

GET /api/calendar?start_date=...&end_date=...&account_id=...

Что реализовать полностью:

Список реестров с колонками: ID, дата, статус, количество платежей, общая сумма, создатель, утверждающий.

Кнопка «Создать реестр» → модалка:

Выбор даты реестра.

Возможность выбрать платежи и нажать «Сформировать реестр».
После создания – редирект на просмотр реестра или обновление списка.

Кнопка «Экспорт CSV» – скачивание файла (в моке можно сгенерировать CSV на фронте, но лучше просто открыть ссылку на эндпоинт).

Заглушки:

Массив реестров с разными статусами.

Функции: getAll, getOne, create (принимает { registry_date, payment_ids }), pay (оплата), export (возвращает ссылку на CSV или Blob).

Эндпоинты для замены:

5. Как организовать переключение между моками и реальным API
Добавить в корень проекта файл .env:

text
REACT_APP_USE_MOCK=true
При сборке для продакшена (или когда бэк готов) – установить REACT_APP_USE_MOCK=false.

Все сервисы должны использовать переменную USE_MOCK для выбора реализации.

Пример api/index.js:

javascript
import { authService } from './endpoints/auth';
import { accountsService } from './endpoints/accounts';
// ... и так далее

export const api = {
  auth: authService,
  accounts: accountsService,
  counterparties: counterpartiesService,
  items: itemsService,
  payments: paymentsService,
  incomes: incomesService,
  calendar: calendarService,
  registries: registriesService,
  reports: reportsService,
  users: usersService,
  audit: auditService,
};
В компонентах используем import { api } from '../api';.

6. Что бэкендеры должны сделать для интеграции
Реализовать все эндпоинты, перечисленные выше, строго по тому формату данных, который используется в моках.

Убедиться, что поля называются так же, как в моках (например, planned_date, closing_balance, has_cash_gap).

Обеспечить поддержку фильтров через query-параметры (период, счёт, статус и т.д.).

Отдавать ошибки в формате { message: '...', errors: { field: ['...'] } } для валидации (422).

Предоставить документацию (OpenAPI или хотя бы список эндпоинтов).

После того как бэкенд готов, фронтенд меняет .env на REACT_APP_USE_MOCK=false и тестирует интеграцию.

7. Советы по разработке заглушек
Имитация задержки: все мок-функции должны возвращать промис с setTimeout (~300 мс), чтобы интерфейс показывал спиннеры.

Имитация ошибок: можно добавить случайный процент ошибок (например, 10% запросов падают с 500), чтобы проверить обработку ошибок.

Пагинация и фильтрация: в моках можно реализовать простую фильтрацию по параметрам (например, если передан status, отфильтровать массив).

Генерация данных: используйте библиотеку faker для создания реалистичных названий, сумм, дат.

8. Заключение
В итоге вы имеете:

Полностью рабочий интерфейс (все страницы, формы, таблицы, фильтры, кнопки).

Всю бизнес-логику (валидацию, переходы статусов, условный рендеринг).

Заглушки, которые эмулируют поведение бэкенда.

Бэкендеры, глядя на готовый фронт и документацию, могут спокойно реализовывать API, не беспокоясь о том, что фронт «упадёт». А вы, в свою очередь, не ждёте их — вы уже сделали свою часть и готовы к интеграции.

Главное правило: все вызовы API инкапсулированы в сервисах, переключение между моками и реальными запросами — это дело одной переменной окружения. Когда придёт время, вы просто переключите флаг — и всё заработает.
